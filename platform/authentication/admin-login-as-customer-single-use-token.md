## Evidence (legacy stack)

### sleek-back

- `controllers/admin/user-controller.js` — `POST /admin/users/get-single-use-authentication-token-dashboard` (`accessControlService.can("companies", "read")`); `POST /admin/users/get-single-use-authentication-token-impersonation`; `POST /admin/users/get-authtoken` (returns `auth_token`); `POST /admin/users/partner-get-sut` (partner).
- `schemas/admin-auth-tokens.js` — `single_use_token`, expiry, `for_user`, `requester`, `company`.
- `services/user-service.js` — `loginWithSingleUseToken` (validate active SUT).
- `controllers/user-controller.js` — body `single_use_token` login path.

### sleek-website

- `src/utils/api.js` — `getAuthenticationTokenForDashboard`, `loginAsOtherUser`, `getAuthenticationToken` (handles `auth_token` vs `single_use_token` response).
- `src/views/admin/company-overview/index.js`, `src/views/admin/companies/edit/index.js` — open customer URL with `&sut=` from returned token.
- `src/components/new-admin-side-menu.js` — commented pattern for sleek-sign + SUT (historical reference).

### Sleek Auth (target)

- No equivalent capability documented in [hosted-login-oauth-pkce.md](./hosted-login-oauth-pkce.md) or related files yet — **gap** to close when product defines impersonation for the new stack.
